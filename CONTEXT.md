# CONTEXT.md — Environmental Monitoring System

This file briefs AI development agents (Gravity, Copilot, etc.) on the current project state, architectural decisions, and conventions. Read before any development session.

---

## What this system does

Automated environmental microbiological monitoring system for food manufacturing facilities operating under FSSC 22000 v6. Replaces manual spreadsheet records with a structured data flow including:

- Mobile-friendly data entry via Google Forms
- Automatic conformity calculation via Google Apps Script
- Structured database in Google Sheets (6 tabs)
- Trend analysis dashboard in Looker Studio
- Automatic email notifications for non-conformances
- Auto-generated monthly PDF reports

---

## Stack

- Google Apps Script (ES5 — GAS-compatible JavaScript)
- Google Sheets as database
- Google Forms as data entry interface
- Looker Studio as dashboard
- clasp for local development
- VS Code as editor
- GitHub for version control

**Critical constraint:** Google Apps Script does not support ES6 modules (`import`/`export`). Inter-module dependencies are managed via explicit header comments declaring dependencies (Option B pattern).

## Module architecture

```
Code.gs             — entry point, triggers, WebApp
Config.gs           — reads CONFIG tab, exposes constants
SamplingPoints.gs   — reads SAMPLING_POINTS tab
Schedule.gs         — reads/writes SCHEDULE tab
Results.gs          — reads/writes RESULTS tab
Calculations.gs     — pure logic: status, trend, recurrence (no Sheets I/O)
Actions.gs          — reads/writes ACTIONS tab
Notifications.gs    — sends emails (no Sheets I/O)
Report.gs           — generates PDF, saves to Drive
Log.gs              — writes to SYSTEM_LOG tab
Utils.gs            — utility functions, no external dependencies
Dev.gs              — development/testing utilities (not triggered in production)
```

**Key rule:** No module accesses a Sheets tab by name directly. Tab names are defined as constants in `Config.gs` and referenced from there. If a tab is renamed, only `Config.gs` needs updating.

**Pure modules:** `Calculations.gs` and `Notifications.gs` receive all data as function parameters and return results. They never call `SpreadsheetApp` or `MailApp` directly. This isolates business logic from I/O.

## Sheets tab contracts

### SAMPLING_POINTS
Master reference table. Read-only during operation. Only the quality team modifies it.

Columns (in order):
`POINT_ID | Sector | Zone | Area | Limit_MA | Limit_BL | Limit_SAL | Active | Full_Name | Frequency`

- `POINT_ID`: unique key, format `SECTOR-ABBREV-SEQ` (e.g., `SD-04-TQ-MIS-01`)
- `Sector`: full name of the production area (e.g., `Spray Dryer 4`, `Homogeneização`)
- `Zone`: A, B, or C (microbiological risk zones per FSSC 22000)
- `Area`: 1, 2, or 3 (criticality level — 1 = product contact, 2 = near product, 3 = environment)
- `Limit_SAL`: Boolean — TRUE if Salmonella testing is required for this point
- `Active`: Boolean — TRUE for active points. Inactive points are archived, not deleted
- `Frequency`: `Monthly` | `Bimonthly` | `Quarterly` | `Biannual` | `Biweekly`

### SCHEDULE
Generated by script at the start of each year. Never filled manually.

Columns:
`COLLECTION_ID | POINT_ID | Sector | Full_Name | Planned_Date | Actual_Date | Delay_Days | Status | Delay_Justification | Collector`

- `Status`: Planned / Collected / Delayed / Not Collected
- `Delay_Justification`: mandatory when Status = Not Collected

### RESULTS
Immutable log. One row per assay per collection. Never edit existing rows.
Corrections are made by adding a new row with `Correction = TRUE` and `Corrected_ID` pointing to the original.

Columns:
`RESULT_ID | COLLECTION_ID | POINT_ID | Timestamp | Collection_Date | Assay | Result | Unit | Limit | Limit_Percentage | Status | Analyst | Correction | Corrected_ID | Notes`

- `Status`: Conforming / Alert / Non-Conforming / Critical (calculated by script, never manual)
- `Limit_Percentage`: Result / Limit * 100 (calculated)

### ACTIONS
Records all responses to deviations.

Columns:
`ACTION_ID | Origin_Result_ID | Action_Type | Opened_Date | Deadline | Responsible | Description | Resample_Result_ID | Action_Status | Closed_Date | Effective | Notes`

- `Action_Type`: Observation / Resample / CAPA
- `Action_Status`: Open / In Progress / Closed / Overdue
- `Effective`: Boolean — filled after analyzing resample result

### CONFIG
System parameters. Editable by quality team without touching code.
Format: column A = parameter name, column B = value.

Parameters:
```
Alert_Threshold        (default: 70)       % of limit that triggers Alert status
Critical_Threshold     (default: 300)      % of limit that triggers Critical status
Resample_Deadline_Days (default: 5)        Business days for resample after NC
Recurrence_Window_Months (default: 6)      Months window for recurrence check
Trend_Consecutive_N    (default: 3)        Consecutive results for upward trend flag
Email_Primary          your@email.com      Main quality contact
Email_Production       prod@email.com      Production supervisor (receives resample alerts)
Email_Manager_1        mgr1@email.com      NC Recurrence and Critical
Email_Manager_2        mgr2@email.com      Critical and Salmonella only
Active_Year            2025               Current schedule year
Seasonality_Rule_Active FALSE             T4 rule — inactive until 2-year baseline exists
```

### SYSTEM_LOG
Automatic event log. Never edited manually.

Columns:
`Timestamp | Event | Reference_ID | Generated_Status | Triggered_Action | User`

## Core business logic

### Status classification (Calculations.gs)

```javascript
// pct = (result / limit) * 100
// Salmonella is binary: result 0 = Absent, 1 = Present

function calculateStatus(assay, result, limit, alertThreshold, criticalThreshold) {
  if (assay === 'SAL') {
    return result > 0 ? 'Critical' : 'Conforming';
  }
  var pct = (result / limit) * 100;
  if (pct < alertThreshold)    return 'Conforming';
  if (pct < 100)               return 'Alert';
  if (pct < criticalThreshold) return 'Non-Conforming';
  return 'Critical';
}
```

### Upward trend detection (Calculations.gs)

Fetches the last N results for the same POINT_ID + Assay from RESULTS (excluding corrections), in chronological order. Returns trend flag if each result is strictly greater than the previous one.

### Recurrence detection (Calculations.gs)

Fetches Non-Conforming results for the same POINT_ID within the last X months (window from CONFIG). Returns recurrence flag if 2 or more are found.

## Main flow — onFormSubmit

```
1.  Receive form data
2.  Validate required fields (Utils.gs)
3.  Look up point in SAMPLING_POINTS (SamplingPoints.gs)
4.  Calculate status and percentage (Calculations.gs)
5.  Write result to RESULTS (Results.gs)
6.  Update SCHEDULE (Schedule.gs)
7.  Check upward trend and recurrence (Calculations.gs)
8.  If status != Conforming: open action in ACTIONS (Actions.gs)
9.  Send appropriate notification (Notifications.gs)
10. Log event to SYSTEM_LOG (Log.gs)
```

## Coding conventions

- Public functions: camelCase, verb + noun (e.g., `calculateStatus`, `findPoint`)
- Private functions: underscore prefix (e.g., `_validateAssay`)
- Constants: UPPER_SNAKE_CASE
- Every function has a block comment: what it does, parameters, return value
- No magic numbers: all numeric values come from CONFIG or named constants
- All log messages in English for consistency

---

## What does NOT go in this repository

- Real company names, facility names, or addresses
- Real employee names or email addresses
- Real sampling point configurations or limit values
- Any data from actual monitoring results

All real operational data lives exclusively in the Sheets file (CONFIG and SAMPLING_POINTS tabs), which is not version-controlled.

Use `config.example.json` as a reference for the CONFIG tab structure without any real values.

## Known gotchas

### UTC date serialization (fixed in v0.2.1)

Google Apps Script serializes `Date` objects in UTC when passed to `appendRow()`.
In UTC-3 (America/Sao_Paulo), midnight local time becomes 21:00 of the previous day
in UTC. Two safeguards are in place:

1. **Code.gs `_extractFormData()`** — Parses dates with `setHours(12, 0, 0, 0)` to
   prevent timezone offsets from crossing the day boundary. Handles both `DD/MM/YYYY`
   and `YYYY-MM-DD` (ISO format from Google Forms Date picker).

2. **Results.gs `writeResult()`** — Formats `collectionDate` as `DD/MM/YYYY` string
   via `formatDate()` before `appendRow()`, bypassing Sheets UTC serialization entirely.

**Rule:** Never pass a raw `Date` object representing a calendar date (without time
significance) directly to `appendRow()`. Always format as string first.

### Biweekly frequency not yet supported in Schedule.gs (v0.3.0)

The `_calculatePlannedDates()` function in `Schedule.gs` accepts 4 frequencies:
`Monthly`, `Bimonthly`, `Quarterly`, `Biannual`. The SAMPLING_POINTS migration
(v0.3.0) introduced 4 points with `Biweekly` frequency (EXT-I/II BOM-SUC and PAS).
The `Biweekly` case must be added to the switch before generating the annual schedule,
otherwise it will throw an unrecognized frequency error.
